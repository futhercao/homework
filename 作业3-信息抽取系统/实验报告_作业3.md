# 灾害信息抽取系统 实验报告(作业3)

## 摘要

本实验构建了一个面向**真实灾害新闻**的信息抽取系统,从报道中抽取 **7 个灾情要素**(灾害类型、发生时间、发生地点、伤亡情况、经济损失、响应级别、救援信息),
组装为结构化事件并构建**知识图谱**。核心创新为 **集成抽取(Ensemble)**:正则 + 触发词 + 命名实体识别(NER)**加权投票**,相比纯正则信息完整度提升约 **4.8%**。
在多媒体方面,系统先用**视觉大模型(Qwen-VL)对配图做质检**、剔除 logo/二维码/UI 等无关图(91 张配图中剔除 71 张),再对保留的真实灾情内容图用 **EasyOCR** 识别文字并回灌进抽取;并提供**逐字段人工评价**界面。
语料由自研**异步爬虫**抓取,共 **118 篇**真实灾害报道(覆盖 **19 个域名 / 14 个权威来源**,全部为政府/通讯社等权威源),**全流程纯 CPU 可运行,无任何编造数据**。

## 一、系统概述与数据真实性

| 项目 | 规模 |
|------|------|
| 灾害文档 | **118 篇**(100% 含真实 URL 与日期) |
| 去重域名 / 来源 | **19 / 14** |
| 有效事件(≥关键要素) | **107 篇(91%)** |
| 配图(VLM 质检后内容图) | **20 张 / 12 篇**(其中含 OCR 文本 8 篇) |
| 知识图谱 | **386 节点 / 641 关系 / 105 事件节点** |

**数据真实性保证**:每篇文档保留原始 `url / source / date`,可逐条溯源(如四川省政府 sc.gov.cn、应急管理部 mem.gov.cn、水利部 mwr.gov.cn、人民网、新华网、中国新闻网、澎湃新闻、中国地震局等);
图片路径已规范化为相对路径 `data/images/<文件名>`,**不含机器绝对路径,可整体迁移**;项目**已删除一切数据生成器**,不含伪造文档。

> 灾害类型分布:地震 39、洪水 33、暴雨 25、台风 11、火灾 5、山体滑坡 3、暴雪 2(共 7 类)。
> 来源 Top:四川省政府(sc.gov.cn 22)、应急管理部(22)、水利部(12)、人民网(12)、新华网(10)、中国地震局(8)、中国新闻网(7)、澎湃新闻(7)。**全部为政府/通讯社权威源,且经"灾害事件"过滤(剔除党务/会议/导航等非事件页),契合灾害领域可信度要求**。

## 二、爬虫设计(灾害语料采集)

针对灾害领域定向采集,在通用异步爬虫基础上强化**主题聚焦**与**质量过滤**:

1. **多源种子 + 灾害查询**:覆盖通讯社、央媒党媒、各级政府/应急/气象/水利站点的社会·灾情频道,以多组灾害关键词驱动链接发现(BFS 浅层扩展)。
2. **三重灾害过滤**:`is_disaster_article`(标题/正文灾害特征)+ 正则灾害类型命中 + 真实性校验 + **事件性过滤**(`is_disaster_event`:**标题须含具体灾害事件/响应信号**——震级/启动响应/遇难/受灾/登陆/强降雨/防汛/救灾资金等,并剔除党务/会议/解读/年度报告/导航/人名等机构话术标题),确保入库为**真实灾害事件**而非政府部门的工作动态/党务页。
3. **来源白名单 + 单源配额**:仅保留政府/通讯社权威域名,并对单一来源设上限,强制来源与题材多样性。
4. **异步并发 + 礼貌抓取 + SimHash 去重 + 配图下载**(按 PNG/JPG 文件头校验落盘,为 OCR 提供素材)。

最终获得 **19 域名 / 14 来源**的权威灾害语料,**来源权威性**(全部政府/通讯社)与**事件纯度**(经事件性过滤)为本系统重点。

## 三、信息抽取方法

### 3.1 七要素与抽取目标

| 要素 | 说明 | 主抽取器 |
|------|------|----------|
| disaster_type 灾害类型 | 地震/台风/洪水/暴雨/火灾… | 正则 + 触发词 |
| event_time 发生时间 | 日期/时段 | 正则 + NER(TIME) |
| event_location 发生地点 | 省市县/震中 | NER(LOC)+ 正则 |
| casualties 伤亡情况 | 死亡/受伤/失踪人数 | 正则 |
| economic_loss 经济损失 | 直接经济损失金额 | 正则 |
| response_level 响应级别 | Ⅰ–Ⅳ级应急响应 | 正则 |
| rescue_info 救援信息 | 转移/搜救/救灾力量 | 触发词 + NER(ORG) |

### 3.2 创新点:集成抽取(Ensemble + 加权投票)

单一正则**召回不足**(灵活表述漏抽)且**精度受限**。本系统融合三个互补抽取器,按字段加权投票:

- **正则引擎(权重基线 0.6)**:模式精确,主攻强格式字段(响应级别、伤亡数字、经济损失金额);
- **触发词引擎**:基于触发词 + 上下文窗口抽取,补正则覆盖不到的自然语言表述;
- **NER 引擎(jieba 词性标注实体)**:用 `jieba.posseg` 标注人名(nr)、地名(ns)、机构(nt)、时间(t),并以领域规则补全完整行政区划与救援机构,补地点、机构、时间字段(若环境装有百度 LAC 则自动改用 LAC 增强)。

**投票流程**:对每个字段收集各抽取器候选 → 按归一化/子串包含关系合并同义候选 → 取累计权重最高者;
对 NER 噪声(如"当日""日从"等前缀)用规则清洗,并对**发生地点做行政区划规范化**(省份锚定抽取、剥离"家住/在/造成"等非地名前缀、合并重复行政后缀、剔除"重点地区/南方地区"等泛化表述,确保只输出具体地点)。每个字段输出 **集成置信度 + 来源标注**(如 `regex+ner`),网页可视化。集成置信度 = 胜出取值那组的权重 ÷ 所有给出候选的抽取器权重之和:例如正则与 NER 都给同一地点则为 **1.0**;若二者给出不同地点(如"积石山县"vs"青海"),按权重 0.45 与 0.40 摊分、约 **0.53**,提示该字段有分歧、需人工复核。页面据此给字段着色(绿≥0.8 / 橙≥0.5 / 灰更低)。

**消融实验**:在相同语料上,集成法相对纯正则平均填充字段数由 **3.70 → 3.88(+4.8%)**,提升集中在**发生地点、救援信息、伤亡情况**等"软"字段(NER 补地名/机构、触发词补自然语言表述;尤其发生地点在正则被规范化严格过滤后,主要靠触发词/NER 召回干净候选);响应级别、灾害类型等强格式字段则正则已能高精度命中,集成主要起**置信度加权与去噪**作用。

### 3.3 事件组装与知识图谱

抽取结果组装为结构化事件(含严重程度定级、信息完整度、缺失字段、摘要);
将事件聚合为**知识图谱**:节点含 事件(105,即 107 篇有效事件中 105 篇要素充足、成功建立事件节点)、地点、灾害类型、机构、时间等,关系连接事件与其要素,共 **386 节点 / 641 关系**,
前端用 ECharts 力导向图展示,支持按类型/地区态势分析。

## 四、多媒体信息抽取(VLM 图片质检 + OCR)

灾害新闻页抓取的"配图"里混着大量与灾情无关的噪声——站点 logo、二维码、UI 图标、广告横幅、纯文字海报等。
若不加甄别直接 OCR,既污染展示,也让"图片文字"里塞满 logo/二维码识别出的乱码。为此配图分两步处理。

### 4.1 VLM 图片质检(剔除无关图)

把每张配图(降采样到 512px)送入视觉大模型(通义千问 `qwen3-vl-plus`,经 DashScope OpenAI 兼容接口),
判定它是否为"有意义的灾害新闻内容图",只输出 `{keep, type, reason}`。判定结果按文件名缓存到
`data/vectors/vlm_verdicts.json`(随交付分发,无密钥也能复现"已清洗"的结果;判定失败则 fail-open 默认保留,
密钥仅取环境变量)。全库 **91 张**配图送检,剔除 **71 张**——网站 logo 25、二维码 10、装饰图 7、UI 图标 6、
无关文字海报 5 等,保留 **20 张**真实灾情内容图(分布于 12 篇文档)。此法与作业2 的图片质检同源,提示词针对灾害题材调整。

> **一个诚实的发现**:权威灾害通报页的"配图"约八成是站点 logo / 二维码 / UI 等页面装饰,而非灾情照片。
> VLM 质检把它们与真实内容图干净地分开,使 OCR 与配图展示不再被噪声淹没。

### 4.2 EasyOCR 识别与回灌

对**通过质检**的内容图,用 **EasyOCR**(`ch_sim+en`,纯 CPU,置信度 ≥0.3)识别文字,以
`content + 【图片文字】<ocr_text>` 拼接进正文一并参与集成抽取。质检后 **8 篇**文档获得真正来自灾情图的 OCR 文本;
结果缓存到 `ocr_cache.json`(演示秒开),OCR 页支持"**实时重新识别**"现场验证非预置。

> **消融对照**:把无关图的 OCR 文本从抽取输入中剔除后,7 要素抽取结果**完全不变**(有效事件 107、平均填充 3.88、
> 各字段填充率均与剔除前一致)——说明灾情要素本就来自正文,而原先"82 篇参与 OCR"里绝大多数是在识别 logo/二维码噪声。
> 剔除后"参与 OCR"由 82 降到 **8**,是更诚实的口径,知识图谱也由 388/643 微调到 386/641 节点/关系。

**关键工程修复**:EasyOCR 底层 `cv2.imread` 无法读取 Windows 中文路径(静默返回空),改用 `PIL.Image` 读图转
`numpy` 数组再喂入,规避中文项目路径下 OCR 全失败的问题。

## 五、实验结果

### 5.1 抽取覆盖率(118 篇)

| 字段 | 填充率 | 说明 |
|------|--------|------|
| 灾害类型 | **118/118 (100%)** | 入库即要求命中灾害类型 |
| 发生地点 | **85/118 (72%)** | NER+正则协同;**经地名规范化,仅保留具体行政区划** |
| 发生时间 | **103/118 (87%)** | 正则多模式 + NER(TIME) |
| 救援信息 | 80/118 (68%) | 触发词 + 机构;权威源重点报道处置/救援力量 |
| 响应级别 | 49/118 (42%) | 启动应急响应的事件含;**支持中文数字"四级"→Ⅳ级 归一化** |
| 伤亡情况 | 14/118 (12%) | 含具体伤亡数字的事件 |
| 经济损失 | 9/118 (8%) | 披露直接经济损失金额的事件 |

平均填充 **3.88/7**,有效事件 **107/118(91%)**。语料经**事件性过滤**后(剔除党务/会议/导航等非灾害事件页,见第二节),
"事件结果"类字段填充率显著高于过滤前——响应级别(42%)、伤亡(12%)、经济损失(8%)均明显提升,因保留的均为**真实灾害事件**而非机构工作动态。
其中伤亡/经济损失绝对值仍偏低,反映**真实新闻在即时通报阶段对具体数字披露有限**(多见于后续跟踪报道),报告如实呈现,非抽取缺陷。

> **发生地点**经**地名规范化**(省份锚定抽取、剥离"家住/在/造成"等非地名前缀、合并重复行政后缀、剔除"重点地区/南方地区"等泛化区域表述),由含泛化噪声的 94% **收敛至 72% 且全部为具体行政区划**——以少量召回换取精度,使人工评价中地点字段准确率显著提升(宁缺毋滥)。

### 5.2 人工评价(逐字段,网页按钮)

抽取结果每个字段旁设 **✅正确 / ⚠️部分 / ❌错误** 按钮,评分者点击后按 1.0 / 0.5 / 0.0 计分,
实时返回本次准确率与逐字段统计并入库(`/evaluate-extraction` 汇总,按字段聚合准确率)。交付件评价库为**干净初始态**,由评分者现场打分生成真实记录。

## 六、对环境与社会可持续发展的影响

- **环境可持续**:异步并发 + 礼貌抓取 + SimHash 去重 + 抽取/OCR/图谱**一次构建持久化复用** + **纯 CPU 友好模型**(EasyOCR、jieba),显著降低算力/能耗。
- **社会可持续**:自动结构化灾情要素、关联事件图谱,可辅助**应急响应与防灾减灾**,服务公共安全;OCR 补全图片中的灾情信息,提升应急信息完整度。

详见系统 `/sustainability` 页面。

---

## 七、局限与展望

- 伤亡/经济损失填充率受**信源披露**限制(权威即时通报少报具体数字),可引入跨文档事件归并、关联后续跟踪报道补全;
- NER 采用 jieba 词性标注 + 领域规则(LAC 可选),灾害领域专名(地名细粒度、救援机构)可用领域词典/微调或接入 LAC/HanLP 等增强;
- 知识图谱可进一步做**事件去重与时序链**,支撑同一灾害的多报道聚合。
