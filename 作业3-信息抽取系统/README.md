# 灾害信息抽取系统(作业3)

从**真实灾害新闻**中抽取结构化事件信息的系统:**集成抽取**(正则 + 触发词 + NER 加权投票)7 个灾情要素、
构建**知识图谱**、**EasyOCR 多媒体抽取**(识别配图文字并回灌进抽取)、以及**逐字段人工评价**。
全流程**纯 CPU 可跑,无需 GPU**。

> 灾害语料全部为爬虫真实抓取,**无任何编造/生成数据**。

> 📋 **验收 / 复现**:见 [`验收指南.md`](验收指南.md) —— 每一步操作对应一个作业需求与一个评分档位。

---

## 一、功能与评分点对应

| 模块 | 说明 | 入口 |
|------|------|------|
| 📋 集成抽取 | 抽取 **7 要素**(类型/时间/地点/伤亡/经济损失/响应级别/救援);**创新点**:正则 + 触发词 + NER **加权投票**,每字段给出集成置信度与来源 | `/` |
| 🕸 知识图谱 | 事件 ↔ 地点 ↔ 类型 ↔ 机构 力导向图谱(ECharts),支持按类型/地区态势分析 | `/kg` |
| 🔤 多媒体抽取 | **VLM 图片质检**（Qwen-VL 剔除 logo / 二维码 / UI 等无关图，91→20 张内容图）+ **EasyOCR**（ch_sim+en, CPU）识别保留内容图文字并**回灌进抽取**；支持"实时重新识别"现场验证 | `/ocr` |
| ✅ 人工评价 | 每个字段旁**打分按钮**(✅正确 / ⚠️部分 / ❌错误)→ 逐字段准确率统计 | `/evaluate-extraction` |
| 🌱 可持续发展 | 系统对环境(低能耗)与社会(防灾减灾)可持续的贡献 | `/sustainability` |

### 创新点:集成抽取(Ensemble)

单一正则容易漏抽、错抽。本系统融合三种互补的抽取器,按字段加权投票:

- **正则引擎**:模式精确,负责"响应级别""伤亡数字"等强格式字段;
- **触发词引擎**:基于触发词 + 上下文窗口,补正则覆盖不到的表述;
- **NER 引擎**:`jieba.posseg` 词性标注识别人名/地名/机构/时间 + 领域规则补全,补地点与机构字段(若装有百度 LAC 则自动改用 LAC 增强)。

投票时按字段权重合并候选(归一化/包含关系去重,取最高权重),并输出**置信度 + 来源标注**,
实测相比纯正则**信息完整度提升 ~4.8%**(平均 3.70 → 3.88 字段/篇)。

## 二、技术架构

```
灾害爬虫 (异步) ──→ 灾害语料 (data/disaster_docs) + 配图 (data/images)
                          │
            ┌─────────────┼──────────────┐
        OCR (EasyOCR)   集成抽取        知识图谱
        图片文字回灌 →  正则+触发词+NER → 事件 → 实体/关系图
                          │
                     7 要素结构化事件 + 逐字段人工评价
```

- **爬虫**:`aiohttp` 异步并发 + UA 轮换 + 单域限流 + `readability` 正文抽取 + **SimHash 去重** + 配图下载(按文件头校验)+ 双重灾害过滤。
- **抽取**:`EnsembleExtractor` 加权投票;命中预计算结果(`extractions.json`)秒级返回。
- **OCR**:EasyOCR 主引擎(纯 CPU),识别结果缓存到 `ocr_cache.json`;图片一律按文件名在本地 `data/images` 解析(可移植、规避 Windows Unicode 路径问题)。
- **知识图谱**:事件聚合为实体(事件/地点/类型/机构)与关系,持久化到 `knowledge_graph.json`。

## 三、安装与使用步骤

### 3.1 环境准备

1. **Python**:建议 3.10 及以上(本项目在 Python 3.13 上验证)。
2. **安装依赖**(纯 CPU,无需 GPU):
   ```bash
   pip install -r requirements.txt
   ```
   `easyocr` 会自动拉取 `torch`、`opencv` 等,首次安装耗时较长。
3. **(可选)VLM 图片质检密钥**:配图质检会调用通义千问视觉模型,密钥仅从环境变量读取:
   ```bash
   setx DASHSCOPE_API_KEY "你的密钥"    # Windows;设置后需重开终端使其生效
   ```
   仓库已附带判定缓存 `data/vectors/vlm_verdicts.json`,**不设置密钥也能复现「已剔除无关图」的结果**。

### 3.2 启动系统

```bash
python run.py web
```
浏览器打开 **http://127.0.0.1:8090**。仓库已附带抽取结果、知识图谱与 OCR 缓存,**开箱即用,无需先爬取或抽取**。

> ⚠️ 作业2 与作业3 默认均使用端口 **8090**,请**勿同时启动**;若端口被占用,先关闭占用进程(见 3.5)。

### 3.3 各功能操作步骤

**(1) 信息抽取 —— 首页 `/`**
1. 页面下方「灾害新闻语料」已列出全部 **118 篇**,可按灾害类型筛选;
2. **点击任意一行文档即可一键抽取**(也可在输入框手输文档 ID,如 `dis_0ef962e3bc6e`);
3. 结果展示 **7 个信息点**(类型 / 时间 / 地点 / 伤亡 / 经济损失 / 响应级别 / 救援),每个字段附**集成置信度**(绿 ≥0.8 / 橙 ≥0.5 / 灰更低)与**来源标注**(如 `regex+ner`);
4. 对每个字段点 **✅ 正确 / ⚠️ 部分 / ❌ 错误**,再点「提交人工评价」,即得本次逐字段准确率;
5. 点「全量抽取 + 知识图谱」可对全部文档批量抽取并重建图谱。
   > 演示建议:挑填充字段较多的文档,如 `dis_0ef962e3bc6e`(积石山地震,6/7 字段)。

**(2) 知识图谱 —— `/kg`**:以 ECharts 力导向图展示「事件 ↔ 地点 ↔ 类型 ↔ 机构」关系,可按类型 / 地区观察态势。

**(3) OCR 多媒体抽取 —— `/ocr`**
1. 左侧「含配图文档」列出经 **VLM 质检后仍含内容图**的文档(logo / 二维码 / UI 等无关图已剔除);
2. 点击一篇,右侧并排显示**原图与识别文本**;
3. 点「实时重新识别」可现场运行 EasyOCR(证明非预置;命中缓存则秒级返回)。

**(4) 抽取评价汇总 —— `/evaluate-extraction`**:按字段聚合的人工评价准确率。

**(5) 可持续发展 —— `/sustainability`**:系统对环境与社会可持续发展贡献的说明。

### 3.4 从零构建语料与抽取结果(可选)

```bash
python run.py crawl --max 400        # 异步爬取真实灾害新闻到 data/disaster_docs
python scripts/filter_images.py      # (可选)VLM 配图质检,生成 data/vectors/vlm_verdicts.json
python run.py ocr                    # 对通过质检的内容图跑 EasyOCR,文字回灌进文档
python run.py extract                # 全量集成抽取 + 构建知识图谱
```
其中 `filter_images.py` 需要 `DASHSCOPE_API_KEY`;若跳过该步,则对全部配图做 OCR(仓库默认已附带质检缓存)。

### 3.5 常见问题

- **端口 8090 被占用 / 与作业2 冲突**:`netstat -ano | findstr :8090` 查看占用进程 PID,`taskkill /F /PID <PID>` 关闭后重启;一次只运行一个系统。
- **页面已打开但「文档列表 / 含配图文档」空白**:多为浏览器缓存所致,按 **Ctrl + Shift + R** 强制刷新。
- **OCR 实时识别较慢**:EasyOCR 为纯 CPU,每图数秒;已识别过的图走缓存秒级返回。从零运行 OCR 时首次会联网下载 EasyOCR 模型(NER 使用 jieba,无需下载)。
- **重新编译实验报告**:报告为 LaTeX,需以 **XeLaTeX** 编译(依赖 ctex 宏包):`xelatex 实验报告_作业3.tex`。

## 五、数据真实性

- `data/disaster_docs/` 内每篇文档均带原始 `url`、`source`、`date`,可逐条溯源至真实新闻页面。
- `data/images/` 为爬虫从新闻页**实际下载**的灾情配图。
- 图片路径已规范化为相对路径(`data/images/<文件名>`),不含任何机器绝对路径,**可整体迁移**。
- 项目**不含任何数据生成器**,不存在编造文档。
